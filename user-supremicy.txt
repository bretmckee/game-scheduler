## What I found

The existing Phase Isolation section already has the right instinct with "modify
code → update all callers in the same phase." The gap is in **three specific
areas**:

### Gap 1: "Tests passing" is too narrow

The implementation prompt template currently says:
> "Before marking any Phase complete... you MUST verify the full unit test suite
> passes."

But pre-commit runs **four gates**, not one:
1. `uv run pytest tests/unit`
2. `uv run mypy shared/ services/`
3. `cd frontend && npm run build`
4. vitest frontend tests

An agent can see pytest pass and consider the phase done, while mypy is about to
block the commit. The planner never explicitly says *"mypy is also a phase
gate."*

### Gap 2: The signature change rule doesn't name the mechanism

The current rule says *"a phase MUST NOT remove or modify code without also
updating every caller."* That's correct, but vague. A planner agent interpreting
"modify" loosely might think *"I changed the service method body, not its
signature."* What really matters is: **if a function's type signature changes,
mypy will report errors on every call site that doesn't change in the same
phase.** Naming mypy explicitly makes this concrete and testable rather than
abstract.

### Gap 3: Forward imports in TDD phases

The current rules don't cover this pattern:
- Phase 1 (RED): Write xfail test that does `from services.bot.announcement_loop
  import AnnouncementLoop`
  - Phase 4: Create `announcement_loop.py`

  At the end of Phase 1, pytest fails to *collect* (not just fail tests) because
  the module doesn't exist. The stub must land in the same phase as the tests
  that import it.

  ---

## Proposed changes

Three concrete additions to task-planner.agent.md:

**A. Rename and expand the phase completion rule** in the Phase Isolation
section to include the full gate list (currently only says "full test suite" —
add mypy as an explicit peer):
> After every phase, ALL of the following must pass: `uv run pytest tests/unit`,
> `uv run mypy shared/ services/`, and (if frontend files changed) `cd frontend
> && npm run build` and `npm run test`. Mypy failures are build failures, not
> test failures — they are equally blocking.

**B. Make the signature change rule mypy-specific** by adding to the existing
"Ordering Rule" section:
> Changing a function, method, or type annotation is a "modification" under this
> rule. If phase N changes any signature, every call site that mypy would flag
> must be updated in the same phase. A plan that relies on mypy errors being
> "fixed in the next phase" is invalid.

**C. Add a forward-import prohibition** as a new sub-rule:
> A phase MUST NOT add imports (in test or production code) for modules,
> classes, or functions that don't exist until a later phase. TDD stubs must be
> created in the same phase as the tests that import them. An import of a
> not-yet-existing module causes pytest collection failure for the entire test
> file, which fails the pre-commit gate.
